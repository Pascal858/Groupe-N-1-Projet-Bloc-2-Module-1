
# Analyse de code – Bloc 2, Module 1

## Contexte
Dans ce module intitulé *« Le code que tu n’as pas écrit »*, nous avons reçu de l'académie, un petit projet développé par un stagiaire. Le code fonctionne à moitié, il n’est pas documenté et les noms de variables ne sont pas très parlants.  
Notre mission n’était pas de le corriger, mais de **le comprendre, l’analyser et le documenter** pour que l’équipe DevStart Agency puisse le reprendre plus facilement.

---

## Notre démarche

### Phase 1 – Exploration
Nous avons commencé par lire le code en entier, sans rien modifier.  
Chacun a pris des notes et ajouté des commentaires pour expliquer, avec ses propres mots, ce que faisait chaque fonction ou section. Nous avons aussi listé les variables et essayé de deviner leur rôle à partir de leur utilisation.

### Phase 2 – Analyse critique
Une fois le code compris, nous avons pris du recul pour identifier ce qui était bien fait et ce qui posait problème.  
Nous avons relevé des soucis de lisibilité, des fonctions trop longues, des noms de variables peu clairs et un manque de commentaires. Nous avons proposé des améliorations, sans toucher au code, uniquement sous forme de suggestions.

### Phase 3 – Rapport et présentation
Enfin, nous avons rédigé un rapport structuré (5 à 8 pages) et préparé une présentation orale de 10 minutes.  
Le rapport décrit le code, détaille notre analyse fonction par fonction, liste les problèmes rencontrés et propose des pistes d’amélioration. La présentation permet de partager notre démarche et de montrer un extrait du code annoté en direct.

---

## Organisation du dépôt
- `/original` → le code fourni, inchangé  
- `/annotated` → le même code, mais avec nos commentaires  
- `README.md` → ce document qui explique notre démarche  
- `rapport.pdf` → notre rapport complet  
- `slides/` → les supports de présentation

---

## Règles de travail
Nous avons travaillé en groupe rotatif : chacun a pris une partie du code mais tout le monde a relu l’ensemble.  
Nous avons utilisé un outil de communication (WhatsApp/Discord) et un tableau de suivi (Notion/Trello).  
Les commits ont été faits régulièrement avec des messages clairs pour retracer notre progression.  
Et surtout, nous avons respecté la consigne : **ne pas réécrire le code**, uniquement l’analyser et le documenter.

---

## Livrables
- Rapport PDF (5–8 pages)  
- Dépôt GitHub public avec code original et annoté  
- Présentation orale de 10 minutes (+ 5 min de questions)

---

## Conclusion
Ce projet nous a montré que comprendre du code écrit par quelqu’un d’autre est une compétence à part entière.  
Nous avons appris à lire, analyser et expliquer un projet technique de manière claire et utile, aussi bien pour des développeurs que pour des personnes moins techniques.
